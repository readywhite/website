# Ready White Agent Instructions

This repository is an AI-native operational platform for Ready White, not just a website.

## Operating cadence

- Run systems checks on all operational systems at 00:00, 12:00, and 18:00 EST daily.
- Before implementing changes, check for operational bottlenecks, identify stale workflows, protect scalability, and preserve package standardization.
- Do not build application logic unless explicitly requested; documentation scaffolds should stay structured, operational, and implementation-ready.

## Optimization priorities

Always optimize for:

- Speed-to-lead.
- Operational reliability.
- Recurring property-management relationships.
- Standardized workflows.
- Reduced vacancy turnover time.
- Pipeline integrity.
- Fast execution.
- Scalable automation.
- Cash flow, close rate, recurring revenue, and operational leverage.

Never optimize for vanity features.

## Ready White operating model

Ready White is not a traditional painting company. Ready White is a standardized property-turnover operating system using subcontractor fulfillment.

Core model:

- Fast-turn occupancy repainting.
- Standardized room pricing.
- AI-assisted estimating.
- GoHighLevel-driven customer pipeline.
- Vendor dispatch workflows.
- Railway API backend.
- GitHub as operational memory.

## Architecture standards

Never redesign the stack unnecessarily.

- Squarespace is the marketing layer.
- Railway is the backend orchestration layer.
- GoHighLevel is the CRM and automation system.
- GitHub is the operational memory, documentation source, and change record.

Always preserve:

- Deterministic scripts.
- Stable naming conventions.
- Standardized GoHighLevel objects.
- Operational documentation.
- Environment-variable security.
- Production-safe architecture.

## Business model standards

Optimize for:

- Package-based quoting.
- Photo-based scope verification.
- Vendor standardization.
- Recurring property-manager relationships.
- Operational speed.
- Automation reliability.

## Customer photo policy

Require the following before quote finalization:

- 1 wide photo of each room.
- 1 photo of the worst wall in each room.
- Photos of ceilings, trim, stains, peeling paint, water damage, smoke damage, holes, and heavy prep areas when present.

Use customer photos to:

- Prevent scope drift.
- Protect margins.
- Detect exception jobs.
- Preserve quote speed.

## Vendor policy

Do not build around flat hourly labor.

Use:

- Preset package buy rates.
- Standard size bands.
- Standard add-ons.
- Exception escalation workflows.

Hourly or change-order rates are only for:

- Severe prep.
- Water damage.
- Smoke or stain blocking.
- Custom repairs.
- Approved scope exceptions.

Always enforce:

- Vendor scorecards.
- Response SLAs.
- Photo proof requirements.
- Standardized scopes.
- Callback tracking.
- Operational redundancy.

## Workflow and automation change policy

When adding or changing workflows, explain:

- ROI impact.
- Operational impact.
- Scalability impact.
- Risk reduction.

Always update the relevant operational docs, SOPs, outreach YAML, audit scripts, KPI reporting, and GoHighLevel object standards when implementing workflow or automation changes.

If a workflow can be standardized, automate it. If a workflow creates operational drift, constrain it.
